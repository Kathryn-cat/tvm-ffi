# TVM-FFI Text Format: Printer/Parser Design

## Goal

TVM-FFI already provides `@tvm_ffi.dataclasses.py_class`, which lets
users define MLIR-style IRs as pure-Python dataclasses — with fields,
structural equality, reflection, and C++/Rust interop for free.

The missing piece: **text format.**  Today, every IR that wants
human-readable printing or source-level parsing must hand-write hundreds
of lines of printer/parser code per node type, duplicating structural
information that is already declared in the dataclass fields.

This document proposes a system that:

1. **Auto-generates printer and parser** for any `@py_class` IR.  A
   typical IR node needs only a one-line trait decorator
   (e.g., `@ffi.traits.For(region=...)`) to get both `__ffi_text_print__`
   and `__ffi_text_parse__` for free.

2. **Allows full customization** for genuinely irregular syntax.  Any
   class can override `__ffi_text_print__` / `__ffi_text_parse__`
   manually, or use `TraitField` escape hatches and `DialectHandlers`
   callbacks for fine-grained control without abandoning the framework.

The target is to **match or outcompete** both TVM's existing TVMScript
infrastructure (~5000 lines of hand-written printer + parser per dialect)
and MLIR's ODS-generated text format, while being pure Python and
requiring near-zero boilerplate from the IR author.

## The "Role Model" Text Format

The text format defines an **abstract "role model" IR** — the set of
Python-syntax patterns that the printer/parser natively understands.
Every user-defined IR maps its concrete fields to these patterns via
traits.

### Functions

A function consists of decorators, attributes, typed parameters, and a
body region:

```python
@tvm_ffi.text.lang.func(
    "${func-kind}",                    # e.g. "tir.prim_func", "relax.function"
    $attr_key0=$attr_val0,             # attrs are kwargs
    $attr_key1=$attr_val1,
)
def ${symbol_name}(
    $arg0: $ty0,
    $arg1: $ty1,
):
    ${region}
```

### For Loop

Similar to MLIR's `scf.for`.  The loop binds one or more induction
variables via a range function.  Attributes reside in the range function
as kwargs:

```python
for $iv0, $iv1, ... in $range_fn(
    [$start,]                          # like Python's range:
    $stop,                             #   range(stop)
    [$step,]                           #   range(start, stop [, step])
    $attr_key0=$attr_val0,             # attrs are kwargs
    $attr_key1=$attr_val1,
):
    ${region}
```

### With Statement

Binds variables from a context expression, then enters a body region:

```python
with ${def_expr} as $v0, $v1, ...:
    ${region}
```

### Assignment

Binds variables from an expression.  May include a type annotation:

```python
$v0, $v1, ... = ${rhs_expr}
$v0: $ty0 = ${rhs_expr}               # annotated form
```

Augmented assignments (`+=`, `-=`, etc.) are handled by the printer AST's
`OperationAST` mechanism, not by traits.

### If / Else

Conditional with one or two regions:

```python
if ${cond}:
    ${then_region}
else:
    ${else_region}
```

### While Loop

```python
while ${cond}:
    ${region}
```

### Class

A class definition with optional decorators and base classes:

```python
@tvm_ffi.text.lang.cls(
    "${class-kind}",
    $attr_key0=$attr_val0,
)
class ${symbol_name}($base0, $base1, ...):
    ${region}
```

### Return

```python
return ${value}
```

### Binary / Unary Operations

```python
${lhs} + ${rhs}                        # binary: +, -, *, /, //, %, **, <<, >>, &, |, ^, and, or
${lhs} < ${rhs}                        # comparison: <, <=, ==, !=, >, >=
-${operand}                            # unary: -, ~, not
```

### Attribute Access

```python
${obj}.${name}
```

### Subscript / Index

```python
${obj}[${idx0}, ${idx1}, ...]
```

### Expression Statement

A bare expression used as a statement (e.g., a void function call):

```python
${expr}
```

## Printer Programmable Control

The printer maps a user-defined IR to the role-model patterns above.
There are two mechanisms, from simplest to most flexible:

1. **Trait decorators** — declarative mapping from IR fields to the role
   model.  The framework auto-generates the printer/parser.
2. **`__ffi_text_print__`** — manual override for full control.

> **Naming:** `__ffi_text_print__` replaces the previous `__ir_print__`.
> There is no backward compatibility with `__ir_print__` — all existing
> uses should be renamed.

### `__ffi_text_print__`: Manual Override

When traits are insufficient (e.g., highly irregular printing logic),
a class can implement `__ffi_text_print__` directly:

```python
@py_class("my_ir.SpecialNode")
class SpecialNode(Node):
    def __ffi_text_print__(self, printer, path):
        """Full manual control over printing."""
        # Build and return AST nodes directly
        return ast.Call(ast.Id("my_ir.special"), [...])
```

If a class has both trait decorators and `__ffi_text_print__`, the manual
method takes precedence — traits are ignored for that class.

### Trait Decorators

Traits are the primary mechanism.  They are applied as decorators stacked
above the `@py_class`:

```python
@py_class("my_ir.Add")
@ffi.traits.BinOp(lhs="field:lhs", rhs="field:rhs", op="+")
class Add(Expr):
    lhs: Expr
    rhs: Expr
```

### `TraitField`: Mapping User IR to the Role Model

Each trait parameter maps a slot in the role-model pattern to a field
(or computed property) on the user's IR class.

```
User IR field  ──TraitField──→  Role-model value  ──printer──→  text
User IR field  ←─TraitField──  Role-model value  ←─parser───  text
```

`TraitFieldLike` has two forms:

```python
TraitFieldLike = str | TraitField
```

**1. `"field:{name}"` string** — identity mapping.  The user's field IS
the role-model value:

```python
# "field:body" → getattr(obj, "body")
Func(symbol="field:name", kind="my_ir.func")
```

**2. `TraitField` object** — explicit conversion when the user's
representation differs from the role model:

```python
@dataclass(frozen=True)
class TraitField:
    getter: str              # user IR → role-model value (for printing)
    setter: str              # role-model value → user IR (for parsing)
```

The `getter` and `setter` strings use prefixed notation:
- `$method:Name` — a method on the IR class (instance method for getter,
  classmethod for setter)
- `$global:Name` — a globally registered function

Example — the user stores `dtype: str` but the role model expects a
printable type object:

```python
@ffi.traits.Value(
    name="field:name_hint",
    type=TraitField(getter="$method:printed_type", setter="$method:parse_type"),
)
class PrimVar(PrimExpr):
    name_hint: str
    dtype: str                     # user representation

    @property
    def printed_type(self):
        """getter: user dtype → role-model type for printing."""
        return self.dtype

    @classmethod
    def parse_type(cls, annotation):
        """setter: role-model annotation → user dtype for parsing."""
        return str(annotation)
```

### `Region`: The Universal Scope

Every region-capable pattern (function body, loop body, with body) is
modeled by a `Region` sub-dataclass:

```python
@dataclass(frozen=True)
class Region:
    body: TraitFieldLike                       # field → list of body statements
    def_values: TraitFieldLike | None = None   # field → Value(s) to define before body
    def_expr: TraitFieldLike | None = None     # field → expression that binds def_values
    ret: TraitFieldLike | None = None          # field → return / output value
    print_pre_hook: str | None = None          # method name called before body printing
    print_post_hook: str | None = None         # method name called after body printing
```

`def_expr` unifies for-loops and with-statements:
- `for x in expr:` → `def_values` = loop var, `def_expr` = iterator expression
- `with expr as x:` → `def_values` = as-var, `def_expr` = context expression
- `def f(params):` → `def_values` = params, no `def_expr` (params in signature)

`print_pre_hook` / `print_post_hook` are method names on the IR class,
called during printing to emit extra statements before/after the body.
This handles patterns like TIR's `match_buffer` and `func_attr`:

```python
@T.prim_func
def matmul(a: T.handle, b: T.handle):
    A = T.match_buffer(a, (128, 128), "float32")   # ← print_pre_hook
    T.func_attr({"global_symbol": "matmul"})        # ← print_pre_hook
    for i, j in T.grid(128, 128):                   # ← body
        ...
```

The printer handles all regions with a single code path:

1. Push a frame, define `def_values` variables
2. Call `print_pre_hook` (if set)
3. Print `body` statements
4. Handle `ret`
5. Call `print_post_hook` (if set)
6. Pop the frame

### Trait Catalog

#### `Value` — SSA Value

The fundamental named value with optional type.  Printed as `name` at
use site, `name: type` at def site.

```python
@dataclass(frozen=True)
class Value:
    name: TraitFieldLike                   # field → display name
    type: TraitFieldLike | None = None     # field → type at def site
```

#### `Func` — Function Definition

```python
@dataclass(frozen=True)
class Func:
    symbol: TraitFieldLike                 # field → function name
    region: Region                         # the function body region
    kind: str | None = None                # decorator string (literal)
    attrs: TraitFieldLike | None = None    # field → attributes dict
    dialect: str | None = None             # body dispatch context (literal)
```

Sugaring options for the decorator:

```python
ffi.traits.Func(
    symbol="field:name",
    region=ffi.traits.Region(
        def_values="field:params",
        body="field:flat_body",
        ret="field:ret_val",
        print_pre_hook="print_prologue",
    ),
    kind="my_dialect.func",
    # sugaring controls:
    # - kind: the decorator function name
    # - attrs: where to print attributes
    #     "field:attrs" → in decorator kwargs
    #     or use print_pre_hook to emit as a separate statement
)
```

#### `For` — Structured Loop

```python
@dataclass(frozen=True)
class For:
    region: Region                         # must have def_values + def_expr
```

Example:

```python
@py_class("my_ir.For")
@ffi.traits.For(
    region=ffi.traits.Region(
        def_values="field:loop_var",
        def_expr="field:iter_expr",        # expression after `in`
        body="field:body",
    ),
)
class For(Stmt):
    loop_var: Var
    iter_expr: Expr          # e.g., my_ir.serial(0, 1024)
    body: list[Stmt]
```

#### `While` — While-Loop

```python
@dataclass(frozen=True)
class While:
    cond: TraitFieldLike                   # field → condition expression
    region: Region                         # loop body
```

#### `If` — If-Then-Else

```python
@dataclass(frozen=True)
class If:
    cond: TraitFieldLike                   # field → condition expression
    then_region: Region                    # true-branch
    else_region: Region | None = None      # optional false-branch
```

#### `With` — Context Manager

```python
@dataclass(frozen=True)
class With:
    region: Region                         # must have def_values + def_expr
    kind: str | None = None                # optional dispatch key (literal)
```

#### `Class` — Class Definition

```python
@dataclass(frozen=True)
class Class:
    symbol: TraitFieldLike                 # field → class name
    region: Region                         # class body
    kind: str | None = None                # decorator string (literal)
    bases: TraitFieldLike | None = None    # field → base classes
```

#### `BinOp` / `UnaryOp` — Operations

```python
@dataclass(frozen=True)
class BinOp:
    lhs: TraitFieldLike                    # field → left operand
    rhs: TraitFieldLike                    # field → right operand
    op: str                                # operator symbol: "+", "-", "and", etc.

@dataclass(frozen=True)
class UnaryOp:
    operand: TraitFieldLike                # field → the operand
    op: str                                # operator symbol: "-x", "~x", "not x"
```

#### `Assign` — Assignment

```python
@dataclass(frozen=True)
class Assign:
    lhs: TraitFieldLike                    # field → target
    rhs: TraitFieldLike                    # field → value
    annotation: TraitFieldLike | None = None  # field → type annotation
```

#### `StmtWithRegion` — Assignment + Scoped Body

The role model for a let-binding: an assignment whose scope contains a
body region.

```python
@dataclass(frozen=True)
class StmtWithRegion:
    lhs: TraitFieldLike                    # field → assignment target
    rhs: TraitFieldLike                    # field → assignment value
    region: Region                         # scoped body
    annotation: TraitFieldLike | None = None
```

#### Other Traits

```python
class Attr:                                # x.y
    obj: TraitFieldLike
    name: TraitFieldLike

class Index:                               # x[i]
    obj: TraitFieldLike
    indices: TraitFieldLike

class Return:                              # return x
    value: TraitFieldLike

class Region:                              # standalone bare region (also a sub-dataclass)
    ...                                    # (defined above)
```

Leaf expressions (`Literal`, `Call`, `Tuple`, `List`, `Dict`, `Slice`)
are handled natively by the printer/parser AST and do not need traits.

### Example: TVM TensorIR

@py_class("tir.PrimFunc")
@ffi.traits.Func(

class PrimFunc(Node):

        for var in self.params:

                frame.stmts.append(printer(MatchBufferRegion(buf, var), ...))

## Parser Finer-Grained Control

### The Problem with Current Dispatch

TVM's current parser uses **two-level dispatch**:

1. **Coarse**: `(dialect_token, ast_node_type)` → handler.
   E.g., `("tirx", "For")`, `("relax", "Assign")`.
2. **Ad-hoc**: Inside the handler, `isinstance` chains on evaluated values.
   E.g., `bind_assign_value` checks `isinstance(value, Frame)`,
   `isinstance(value, Buffer)`, etc.

Problems:
- Every dialect must register handlers for every AST node type.
- Mixed-dialect IR (Relax calling TIR) needs a token stack.
- `isinstance` chains grow linearly with supported types.
- Adding a new construct requires touching the dispatch table AND the
  isinstance chains.

### Value-Driven Dispatch: `__ffi_text_parse__`

Instead of dispatching on `(dialect, ast_type)`, we **evaluate first**
and let the returned value handle parsing:

```
Parser sees:    with T.block("name") as [vi, vj]: ...
                     ^^^^^^^^^^^^^^^^
Step 1: Evaluate    T.block("name")  →  returns BlockFrame object
Step 2: Call        BlockFrame.__ffi_text_parse__(parser, with_node)
                    ↑ the object itself knows how to parse the `with` body
```

The protocol:

```python
def __ffi_text_parse__(self, parser: IRParser, node: ast.AST) -> Any:
    """Parse the surrounding AST context using this evaluated value.

    Parameters
    ----------
    self : The evaluated expression result (e.g., ForFrame, BlockFrame,
           decorator, relax.Expr)
    parser : The parser, providing var_table, eval_expr, visit_stmt
    node : The surrounding Python AST node (ast.For, ast.With,
           ast.Assign, ast.FunctionDef, etc.)

    Returns
    -------
    The constructed IR node.
    """
```

### Complete Case Analysis

Below is every Python AST pattern used in TVM's TIRx and Relax dialects,
with how each maps to our design.

#### Value-Driven Cases (via `__ffi_text_parse__`)

These cases evaluate an expression, then dispatch on the returned value:

**For-loop** — `for x in T.serial(0, 128):`

```python
def visit_For(self, node: ast.For):
    iter_val = self.eval_expr(node.iter)       # → ForFrame
    return self._value_dispatch(iter_val, node)
```

Covers: `T.serial`, `T.parallel`, `T.vectorized`, `T.unroll`,
`T.thread_binding`, `T.grid`, and `range()`.  Each returns a Frame whose
`__ffi_text_parse__` handles loop-var binding and body parsing.

**With-statement** — `with T.block("name") as [vi, vj]:`

```python
def visit_With(self, node: ast.With):
    if len(node.items) == 1:
        ctx_val = self.eval_expr(node.items[0].context_expr)
        return self._value_dispatch(ctx_val, node)
    # Multi-item: structural fallback (see below)
```

Covers: `T.sblock`, `T.Assert`, `T.attr`, `T.launch_thread`,
`R.dataflow()`, `R.If`.  The Frame's `__ffi_text_parse__` manages:
- Binding `as`-variables from `__enter__` return
- Parsing the body
- Post-exit processing (e.g., Relax dataflow `output_vars`)

**Assignment** — `A = T.match_buffer(a, ...)` / `vi = T.axis.spatial(128, i)`

```python
def visit_Assign(self, node: ast.Assign):
    if _is_subscript_lhs(node):              # A[i,j] = v → structural
        return self._visit_subscript_store(node)
    rhs = self.eval_expr(node.value)
    return self._value_dispatch_or_bind(rhs, node)
```

Covers: `T.match_buffer`, `T.alloc_buffer`, `T.decl_buffer`, `T.var`,
`T.size_var`, `T.axis.spatial`, `T.axis.reduce`, `T.axis.remap`,
`T.env_thread`, `T.meta_var`, `R.emit(expr)`, `R.match_cast(...)`.
Each returned object's `__ffi_text_parse__` handles binding to the LHS.

For Relax, this replaces the `bind_assign_value` isinstance chain:

```python
# relax.Expr: emit SSA binding
class RelaxExpr:
    def __ffi_text_parse__(self, parser, node):
        var = R.emit(self)
        parser.var_table.define(node.targets[0].id, var)

# MatchCastPair: emit match_cast
class MatchCastPair:
    def __ffi_text_parse__(self, parser, node):
        var = R.emit_match_cast(self.value, self.struct_info)
        parser.var_table.define(node.targets[0].id, var)

# Cross-dialect: TIR Var inside Relax
class TIRVar:
    def __ffi_text_parse__(self, parser, node):
        IRBuilder.name(node.targets[0].id, self)
```

**Annotated assignment** — `v: T.int32 = expr`

```python
def visit_AnnAssign(self, node: ast.AnnAssign):
    rhs = self.eval_expr(node.value)
    return self._value_dispatch_or_bind(rhs, node)
    # __ffi_text_parse__ receives ast.AnnAssign → can read node.annotation
```

The `__ffi_text_parse__` method reads annotation from the node:
```python
def __ffi_text_parse__(self, parser, node):
    if isinstance(node, ast.AnnAssign):
        anno = parser.eval_expr(node.annotation)  # e.g., T.int32, R.Tensor(...)
        # Use anno for typed binding
```

**Decorated function** — `@T.prim_func def f(A: T.Buffer(...)):`

```python
def visit_FunctionDef(self, node: ast.FunctionDef):
    for dec in node.decorator_list:
        dec_val = self.eval_expr(dec)
        if result := self._try_value_dispatch(dec_val, node):
            return result
    raise ParseError("Unrecognized function decorator")
```

Covers: `@T.prim_func`, `@T.prim_func(private=True)`, `@R.function`,
`@R.function(pure=True)`, `@I.ir_module`.  The decorator's
`__ffi_text_parse__` handles parameter parsing, symbolic variable
collection, body parsing, and return type processing.

**Expression statement** — `T.evaluate(expr)` / `T.reads(...)` / bare Frame

```python
def visit_Expr(self, node: ast.Expr):
    val = self.eval_expr(node.value)
    if hasattr(val, "__ffi_text_parse__"):
        return val.__ffi_text_parse__(self, node)  # Frame auto-enter, etc.
    # Fallback: discard (docstrings, None, side-effect-only calls)
```

Covers: Frames returned by `T.launch_thread(...)` as bare statements,
`T.evaluate(...)`, `T.reads(...)`, `T.writes(...)`, `T.func_attr({...})`.
Most of these are fire-and-forget calls (evaluated for side effects); the
result is discarded.  If a Frame is returned, its `__ffi_text_parse__`
auto-enters it.

**Class definition** — `@I.ir_module class MyModule:`

```python
def visit_ClassDef(self, node: ast.ClassDef):
    for dec in node.decorator_list:
        dec_val = self.eval_expr(dec)
        if r := self._try_value_dispatch(dec_val, node):
            return r
    raise ParseError("Unrecognized class decorator")
```

The module decorator's `__ffi_text_parse__` implements 3-pass parsing:
1. Process non-function statements (module attrs, global infos)
2. Forward-declare all functions (register GlobalVars)
3. Parse function implementations

#### Structural Cases (Parser Built-Ins)

These constructs are always parsed the same way regardless of dialect.
No value-driven dispatch.

**Subscript store** — `A[i, j] = value`

Detected before RHS evaluation: if LHS is a subscript, parse as a store
operation.  Covers buffer stores and augmented buffer stores.

**Augmented assignment** — `A[i] += B[j]`

Desugared by the parser:
1. Evaluate LHS → buffer load
2. Evaluate RHS
3. Construct BinOp(load, rhs) with the operator
4. Store result back

No trait or `__ffi_text_parse__` needed.

**Continue / Break**

```python
def visit_Continue(self, node): ...  # emit continue intrinsic
def visit_Break(self, node): ...     # emit break intrinsic
```

**Pass** — silently ignored.

#### Dialect-Contextual Cases

Some structural constructs (`if`, `while`, `return`) have semantics that
differ between dialects.  For example:
- TIR `if`: condition can be `PrimExpr` (IR-level) or `bool` (static
  branching at parse time)
- Relax `if`: always creates `R.If`/`R.Then`/`R.Else` frames; `else` is
  mandatory; result is bound to a variable

These are handled via a **dialect context** set by the enclosing
function's `__ffi_text_parse__`.  The decorator provides a
`DialectHandlers` object:

```python
@dataclass
class DialectHandlers:
    """Dialect-specific handlers for structural constructs."""
    handle_if: Callable | None = None
    handle_while: Callable | None = None
    handle_return: Callable | None = None
```

The function decorator's `__ffi_text_parse__` sets this:

```python
class PrimFuncDecorator:
    def __ffi_text_parse__(self, parser, node: ast.FunctionDef):
        with parser.set_dialect(TIR_HANDLERS):
            ... parse params, body ...
```

The parser's structural visitors consult the dialect:

```python
def visit_If(self, node: ast.If):
    cond = self.eval_expr(node.test)
    if self.dialect.handle_if:
        return self.dialect.handle_if(self, cond, node)
    # Default: Python-style if (for module-level code)
    ...
```

TIR provides:
```python
def tir_handle_if(parser, cond, node):
    if isinstance(cond, bool):
        # Static branching — parse only the taken branch
        if cond:
            parser.visit_body(node.body)
        elif node.orelse:
            parser.visit_body(node.orelse)
    else:
        # IR-level If with T.If / T.Then / T.Else frames
        ...
```

Relax provides:
```python
def relax_handle_if(parser, cond, node):
    if node.orelse is None:
        raise ParseError("Relax requires else branch")
    # Create R.If / R.Then / R.Else frames, bind result var
    ...
```

This replaces TVM's dispatch tokens for structural constructs while
keeping value-driven dispatch for everything else.

#### Multi-Item `with` (Structural Fallback)

TIR supports `with A as x, B as y:` (multiple context managers).  When
there are multiple items, the parser falls back to structural handling:

```python
def visit_With(self, node: ast.With):
    if len(node.items) == 1:
        ctx_val = self.eval_expr(node.items[0].context_expr)
        return self._value_dispatch(ctx_val, node)
    # Multi-item: enter each frame, visit body once
    with ExitStack() as stack:
        for item in node.items:
            ctx_val = self.eval_expr(item.context_expr)
            result = stack.enter_context(ctx_val)
            if item.optional_vars:
                self._bind_vars(item.optional_vars, result)
        self.visit_body(node.body)
```

### The Parser Core

```python
class IRParser:
    def __init__(self):
        self.var_table = VarTable()
        self.dialect = DialectHandlers()     # set by function decorator

    # --- Value-driven visitors ---

    def visit_For(self, node):
        return self._value_dispatch(self.eval_expr(node.iter), node)

    def visit_With(self, node):
        if len(node.items) == 1:
            return self._value_dispatch(
                self.eval_expr(node.items[0].context_expr), node)
        return self._visit_multi_with(node)  # structural fallback

    def visit_Assign(self, node):
        if _is_subscript_lhs(node):
            return self._visit_subscript_store(node)
        return self._value_dispatch_or_bind(self.eval_expr(node.value), node)

    def visit_AnnAssign(self, node):
        return self._value_dispatch_or_bind(self.eval_expr(node.value), node)

    def visit_FunctionDef(self, node):
        for dec in node.decorator_list:
            if r := self._try_value_dispatch(self.eval_expr(dec), node):
                return r
        raise ParseError("Unrecognized function decorator")

    def visit_ClassDef(self, node):
        for dec in node.decorator_list:
            if r := self._try_value_dispatch(self.eval_expr(dec), node):
                return r
        raise ParseError("Unrecognized class decorator")

    def visit_Expr(self, node):
        val = self.eval_expr(node.value)
        self._try_value_dispatch(val, node)  # Frame auto-enter, or discard

    # --- Structural visitors (dialect-contextual) ---

    def visit_If(self, node):
        cond = self.eval_expr(node.test)
        if self.dialect.handle_if:
            return self.dialect.handle_if(self, cond, node)
        # Default: Python-level branching
        if cond:
            self.visit_body(node.body)
        elif node.orelse:
            self.visit_body(node.orelse)

    def visit_While(self, node):
        if self.dialect.handle_while:
            return self.dialect.handle_while(self, self.eval_expr(node.test), node)
        raise ParseError("while not supported outside IR function")

    def visit_Return(self, node):
        val = self.eval_expr(node.value) if node.value else None
        if self.dialect.handle_return:
            return self.dialect.handle_return(self, val, node)
        raise ParseError("return not supported outside IR function")

    def visit_AugAssign(self, node):  ...  # desugar to load + op + store
    def visit_Continue(self, node):   ...  # emit intrinsic
    def visit_Break(self, node):      ...  # emit intrinsic
    def visit_Assert(self, node):     ...  # delegate to dialect if set
    def visit_Pass(self, node):       pass # no-op

    # --- Dispatch helpers ---

    def _value_dispatch(self, val, node):
        if hasattr(val, "__ffi_text_parse__"):
            return val.__ffi_text_parse__(self, node)
        raise ParseError(f"Expected IR construct, got {type(val)}")

    def _value_dispatch_or_bind(self, val, node):
        if hasattr(val, "__ffi_text_parse__"):
            return val.__ffi_text_parse__(self, node)
        self._default_bind(val, node)

    def _try_value_dispatch(self, val, node):
        if hasattr(val, "__ffi_text_parse__"):
            return val.__ffi_text_parse__(self, node)
        return None
```

### How Traits Generate `__ffi_text_parse__`

The `@ffi.lang` module entries auto-generate `__ffi_text_parse__` on
Frame/proxy objects using trait metadata:

```python
# ffi.lang.for_iter(TIRFor) generates:
class _ForFrame:
    def __init__(self, iter_expr, ...):
        self.iter_expr = iter_expr

    def __ffi_text_parse__(self, parser, node: ast.For):
        """Auto-generated from For trait on TIRFor."""
        with parser.var_table.frame():
            loop_var = parser.create_var(node.target)
            parser.var_table.define(node.target.id, loop_var)
            body = parser.visit_body(node.body)
            return TIRFor(loop_var=loop_var, iter_expr=self.iter_expr, body=body)

# ffi.lang.with_ctx(BlockRealize) generates:
class _BlockFrame:
    def __ffi_text_parse__(self, parser, node: ast.With):
        with parser.var_table.frame():
            # Bind as-variables
            as_vars = self._create_iter_vars()
            if node.items[0].optional_vars:
                parser.bind_vars(node.items[0].optional_vars, as_vars)
            # Parse body
            body = parser.visit_body(node.body)
            # Post-exit processing (e.g., output_vars for dataflow)
            return self._construct_ir(as_vars, body)

# ffi.lang.decorator(PrimFunc) generates:
class _PrimFuncDecorator:
    def __ffi_text_parse__(self, parser, node: ast.FunctionDef):
        with parser.var_table.frame():
            with parser.set_dialect(TIR_HANDLERS):
                params = self._parse_params(parser, node)
                body = parser.visit_body(node.body)
                return PrimFunc(params=params, body=body, ...)
```

### Comparison with TVM's Current System

| Aspect | TVM today | Our design |
|--------|-----------|------------|
| Dispatch table | Global `ParseVTable[(token, type)]` | None |
| Dialect tokens | Stack, must push/pop | `DialectHandlers` set by function decorator |
| `if`/`while`/`return` | Per-dialect dispatch handlers | `DialectHandlers` callbacks |
| `bind_assign_value` | Growing isinstance chain | Each type's `__ffi_text_parse__` |
| Adding a construct | Register in vtable + isinstance case | Add `__ffi_text_parse__` |
| Mixed dialects | Token stack management | Each object is self-contained |
| Parser core size | ~300 lines + per-dialect handlers | ~60 lines + auto-generated `__ffi_text_parse__` |
| AugAssign | Per-dialect handler | Parser built-in (desugar to load+op+store) |
| Subscript store | Per-dialect special case | Parser built-in (detect subscript LHS) |
| Multi-item `with` | Per-dialect ExitStack | Parser built-in fallback |
| Expression stmt | Per-dialect isinstance chain | `_try_value_dispatch` + discard |
| Module 3-pass | Hardcoded in IR parser | Module decorator's `__ffi_text_parse__` |
| GlobalVar patching | Monkey-patch in enter_token | Language module proxy `__call__` |

## Language Modules: Typed eDSL Surfaces

The printed text is valid Python.  For it to be editable with IDE type
checking, a **language module** provides the namespace that makes it
type-checkable (like Triton's `triton.language` or TVM's `T.*`).

### `@ffi.lang` Decorator

Each dialect declares a language module that maps trait `kind` strings
to Python-level API entries:

```python
@ffi.lang("T", dispatch_token="tirx")
class T:
    prim_func      = ffi.lang.decorator(PrimFunc)
    serial         = ffi.lang.for_iter(TIRFor, kind="serial")
    parallel       = ffi.lang.for_iter(TIRFor, kind="parallel")
    block          = ffi.lang.with_ctx(BlockRealize)
    match_buffer   = ffi.lang.binding(MatchBufferRegion)
    alloc_buffer   = ffi.lang.binding(AllocBuffer)
    Buffer         = ffi.lang.proxy(BufferProxy)
    int32          = ffi.lang.dtype("int32")
    float32        = ffi.lang.dtype("float32")
```

Each `ffi.lang.*` descriptor is a **one-liner** that links a name to an
IR type.  The framework uses the trait on that IR type to:
- Generate runtime proxy objects (with `__ffi_text_parse__`)
- Generate `.pyi` stubs via `tvm-ffi-stubgen language`

### Auto-Generated Operator Stubs

The framework scans all `BinOp`/`UnaryOp` trait registrations to generate
operator stubs on the language module's var proxy.

### Dependent Types (Symbolic Shapes)

`R.Tensor((N, M), "float32")` references symbolic variables not yet in
scope.  Language module proxies handle this via deferred evaluation: the
proxy stores symbolic references, and `__ffi_text_parse__` resolves them
against the parser's variable table when invoked.

### Customized `with` Scopes (Frame Objects)

Frame objects are auto-generated from region-capable traits.  The frame's
`__enter__` return type is derived from `Region.def_values`.  Scope-local
entries (constructs valid only inside a `with`) are declared via
`ffi.lang.with_ctx(..., scope={...})`.

## Open Questions

1. **`DialectHandlers` granularity.**
   The current design has `handle_if`, `handle_while`, `handle_return`.
   TIR also needs dialect-specific `handle_assert` (structured error
   messages) and `handle_aug_assign` (buffer AugAssign).  Should
   `DialectHandlers` be open-ended (a dict of handler names), or a
   fixed set of known structural constructs?

2. **Expression evaluator and operator dispatch.**
   TVM's `ExprEvaluator` uses `OpVTable[(operand_type, op_type, index)]`
   for operator overloading during expression evaluation.  Our design
   replaces dispatch tables, but the expression evaluator still needs
   type-based operator dispatch.  Should `BinOp`/`UnaryOp` traits
   auto-register into the expression evaluator's operator table?

3. **Language module stub fidelity.**
   `setter` method signatures may not suffice for overloaded stubs
   (e.g., `T.serial(stop)` vs `T.serial(start, stop, step=...)`).
   Support optional `lang_*` stub-only methods; fall back to `setter`
   when absent.

4. **Trait inheritance.**
   Subclasses must apply trait decorators explicitly.

5. **Scope-local entries at parse time.**
   `T.reads(...)`, `T.writes(...)`, `T.alloc_buffer(...)` are only
   valid inside `T.sblock(...)`.  The `ffi.lang.with_ctx(..., scope={})`
   mechanism declares this statically, but how does the parser enforce
   it at runtime?  One approach: the Frame's `__ffi_text_parse__`
   temporarily adds scope-local entries to the expression evaluator's
   namespace during body parsing.

6. **`T.meta_var` and parse-time-only values.**
   `N, M = T.meta_var(self.shape)` wraps a Python value that exists
   only during parsing (not in the final IR).  The `meta_var` wrapper's
   `__ffi_text_parse__` on Assign can unwrap to the inner value.
   But how should the printer handle these?  They don't appear in the
   IR — they're reconstructed from context during printing.

7. **`T.comm_reducer` and lambda expressions.**
   `T.comm_reducer(lambda x, y: x + y, [T.float32(0)])` passes Python
   lambdas as arguments.  The expression evaluator handles this
   natively (lambdas are just Python objects).  But the printer needs
   to reconstruct the lambda syntax from the IR's combiner function.
   This likely requires a manual `__ffi_text_print__` override.
