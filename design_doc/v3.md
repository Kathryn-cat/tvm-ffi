# TVM-FFI Traits & Text Format Design

## Goal

`@tvm_ffi.dataclasses.py_class` already gives us the structural IR
definition: fields, reflection, equality, and C++/Rust interop.

This design adds:

1. **Semantic traits** — a small vocabulary describing WHAT each IR node IS
   (a loop, a value, a function).  Consumed by all tools.
2. **Auto-generated printer/parser** — for most `@py_class` IR nodes, a
   one-line trait decorator produces both `__ffi_text_print__` and
   `__ffi_text_parse__`.
3. **Clean escape hatch** — genuinely irregular syntax gets a manual
   override without polluting the core.

### Scope

The text format is **Python-style** — printed IR is valid Python that can
be edited with IDE support.  This is the same format family as TVM's
TVMScript.

**Not in scope**: MLIR assembly syntax (`%0 = arith.addi %1, %2 : i32`).
MLIR operations that want MLIR-native text should use MLIR's ODS and
`assemblyFormat`.  The two systems complement each other — a project can
use both.

### Design principles

- **Concise.**  One decorator per class.  No configuration DSLs.
- **Layered.**  Semantic traits are the foundation; `text_*` is one
  consumer.  Other consumers (optimizers, visualizers) read traits and
  ignore `text_*`.
- **Honest.**  The auto-generated printer targets canonical syntax
  recoverable from the final IR.  Sugar is a separate layer.
- **Usable out of the box.**  `@py_class` with zero annotations gets a
  working printer/parser.  Traits and `text_*` are optional.

### Validated against

- TVM TIRx: `3rdparty/tvm/python/tvm/tirx/`, printer in
  `3rdparty/tvm/src/script/printer/tirx/` (7 files, ~2500 LOC), parser
  in `3rdparty/tvm/python/tvm/script/parser/tirx/` (15 dispatch handlers)
- TVM Relax: `3rdparty/tvm/python/tvm/relax/`, printer in
  `3rdparty/tvm/src/script/printer/relax/` (7 files, ~2000 LOC), parser
  in `3rdparty/tvm/python/tvm/script/parser/relax/` (14 dispatch handlers)
- MLIR: `assemblyFormat` + ODS in `/llvm-project/mlir/`, custom
  printer/parser fallback patterns in SCF, Linalg, Func dialects
- cuteDSL: `/dynamic-kernel-generator/DkgDSL/`, Python DSL with operator
  overloading + AST preprocessing targeting MLIR

## Trait Catalog

### Field References

Trait parameters reference IR fields via `$`-prefixed strings:

- **`"$field:name"`** — read a field: `getattr(obj, "name")`
- **`"$method:name"`** — call a method: `getattr(obj, "name")()`
- **`"$global:name"`** — call a registered global function

`$method:` is the standard approach when the printed value is computed
from multiple fields (common for loop bounds, type annotations, etc.).

```python
@ffi.traits.For(
    region=ffi.traits.Region(def_values="$field:loop_var", body="$field:body"),
    start="$field:min",
    end="$method:computed_end",            # = min + extent
    step="$field:step",
    text_printer_kind="$method:kind_prefix",   # T.serial / T.parallel / ...
)
class TIRFor(Stmt):
    @property
    def computed_end(self):
        return self.min + self.extent

    @property
    def kind_prefix(self):
        return {ForKind.SERIAL: "T.serial", ...}[self.kind]
```

### `Region` — Scoped Body

```python
class Region:
    body: str                              # "$field:..." or "$method:..."
    def_values: str | None = None          # vars defined before body
    def_expr: str | None = None            # expression that binds def_values
    ret: str | None = None                 # return / output value
```

`def_expr` is the expression after `in` or `as` in Python:
- `with expr as x:` → `def_values`=as-var, `def_expr`=context expression
- `def f(params):` → `def_values`=params, no `def_expr`

For **for-loops**, use the `For` trait's `start`/`end`/`step` fields
instead of `def_expr` (see below).

### `Value` — SSA Variable

```python
class Value:
    name: str                              # "$field:..." or "$method:..."
    type: str | None = None                # type annotation at def site
```

### `Func` — Function

```python
class Func:
    symbol: str
    region: Region
    attrs: str | None = None
    text_printer_kind: str | None = None   # e.g. "$lang.prim_func"
    text_printer_pre: str | None = None    # e.g. "$method:print_prologue"
```

### `For` — Loop

```python
class For:
    region: Region                         # must have def_values + body
    # Range bounds (the common case for structured loops):
    start: str | None = None               # "$field:..." — default: 0
    end: str | None = None                 # "$field:..." or "$method:..."
    step: str | None = None                # "$field:..." — default: 1
    attrs: str | None = None
    text_printer_kind: str | None = None   # overrides default "range"
```

When `end` is provided, the printer auto-constructs the iterator
expression using Python's `range(start, end, step)` convention, with
default elision:

| Fields set | Printed as |
|------------|-----------|
| `end` only | `range(end)` |
| `start` + `end` | `range(start, end)` |
| `start` + `end` + `step` | `range(start, end, step)` |
| any + `text_printer_kind` | `kind(start, end, step=step, ...)` |

When `end` is NOT provided, the printer falls back to `region.def_expr`
for generic iteration (e.g. `for x in collection:`).

The `(start, end, step)` triple is **semantic** — consumed by all tools:
- Loop optimizers read bounds directly from the trait
- Dependence analysis uses bounds for iteration space reasoning
- The text printer uses bounds to construct the range expression

### `With` — Context Manager

```python
class With:
    region: Region                         # must have def_values + def_expr
    text_printer_kind: str | None = None
    text_printer_pre: str | None = None
    text_printer_post: str | None = None
```

### `While` — While-Loop

```python
class While:
    cond: str
    region: Region
```

### `If` — Conditional

```python
class If:
    cond: str
    then_region: Region
    else_region: Region | None = None
```

### `Class` — Module / Class

```python
class Class:
    symbol: str
    region: Region
    bases: str | None = None
    text_printer_kind: str | None = None
    text_printer_pre: str | None = None
```

### `BinOp` / `UnaryOp`

```python
class BinOp:
    lhs: str
    rhs: str
    op: str                                # "+", "-", "and", etc.

class UnaryOp:
    operand: str
    op: str                                # "-", "~", "not"
```

### `Assign` — Assignment / Binding

```python
class Assign:
    def_values: str                        # target variable(s)
    rhs: str                               # value expression
```

### `Load` / `Store` — Memory Access

```python
class Load:
    source: str                            # buffer/tensor
    indices: str                           # index expressions

class Store:
    target: str
    indices: str

```

### `Return`

```python
class Return:
    value: str
```

### Who Consumes Semantic Traits

| Consumer | What it reads |
|----------|--------------|
| Text printer/parser | Semantic fields + `text_*` parameters |
| Graph visualizer | `Value.name`, `Region` nesting |
| Interpreter | `For.start`/`end`/`step`, `If` condition |
| Loop optimizer | `For.start`/`end`/`step`, `For.attrs` |
| Type checker | `Value.type`, `Func` return |
| Code generator | `BinOp.op`, `Load`/`Store` patterns |
| Dependence analysis | `Load.source`/`Store.target` + indices |

## `text_*` Parameters

Text-format-specific parameters use the `text_` prefix and are ignored
by all non-text consumers.

| Parameter | Purpose | Example |
|-----------|---------|---------|
| `text_printer_kind` | Decorator/callee text | `"$lang.prim_func"` |
| `text_printer_pre` | Method called before body | `"$method:print_prologue"` |
| `text_printer_post` | Method called after body | `"$method:print_outputs"` |

When no `text_*` parameters are provided, the framework infers a default
rendering from the semantic trait alone.

## Customization Levels

### Level 0: Zero-Config Default

Every `@py_class` gets a working printer/parser with no annotations:

```python
@py_class("my_ir.Add")
class Add(Expr):
    lhs: Expr
    rhs: Expr
```

Output: `my_ir.Add(lhs=<lhs>, rhs=<rhs>)`

This is the "debug" format.  Functional but not human-friendly.

### Level 1: Trait Decorators

A trait maps the IR node to a Python syntax pattern.  The framework
auto-generates printer and parser:

```python
@py_class("my_ir.Add")
@ffi.traits.BinOp(lhs="$field:lhs", rhs="$field:rhs", op="+")
class Add(Expr):
    lhs: Expr
    rhs: Expr
```

Output: `<lhs> + <rhs>`

For structural nodes, use `text_*` when the default rendering needs
customization:

```python
@ffi.traits.Func(
    symbol="$field:name",
    region=ffi.traits.Region(def_values="$field:params", body="$field:body"),
    text_printer_kind="T.prim_func",
    text_printer_pre="$method:print_prologue",
)
class PrimFunc(Node):
    def print_prologue(self, printer_ctx, frame):
        """Emit func_attr and match_buffer before the body."""
        if self.attrs:
            frame.add(printer_ctx.call_stmt("T.func_attr", self.attrs))
        for var, buf in self.buffer_map.items():
            frame.add(printer_ctx.assign_stmt(buf.name, "T.match_buffer", var, buf.shape, buf.dtype))
```

### Level 2: Manual Override

For genuinely irregular syntax:

```python
@py_class("my_ir.CommReducer")
class CommReducer(Node):
    def __ffi_text_print__(self, printer_ctx, path):
        """Full control over printing."""
        ...

    @classmethod
    def __ffi_text_parse__(cls, parser_ctx, frame):
        """Full control over parsing."""
        ...
```

`__ffi_text_print__` is an instance method.  `__ffi_text_parse__` is a
classmethod / static factory.  Manual methods take precedence over traits.

## Printer Algorithm

The printer is a **canonicalizer** — it prints the stable syntax defined
by the trait, not historical spelling.

### Dispatch

```python
def print_node(node, ctx, path):
    # Level 2 takes precedence
    if hasattr(node, '__ffi_text_print__'):
        return node.__ffi_text_print__(ctx, path)
    trait = get_trait(node)
    if trait is None:
        return default_print(node, ctx, path)    # Level 0
    return trait_print(node, trait, ctx, path)    # Level 1
```

### Per-Trait Printing

**BinOp / UnaryOp** — inline expression:

```python
def print_binop(node, trait, ctx, path):
    lhs = ctx.print_expr(resolve(trait.lhs, node), path.attr("lhs"))
    rhs = ctx.print_expr(resolve(trait.rhs, node), path.attr("rhs"))
    return ast.BinOp(lhs, trait.op, rhs)
```

**Value** — name at use site, `name: type` at def site:

```python
def print_value_def(node, trait, ctx, path):
    name = resolve(trait.name, node)
    var_id = ctx.define_var(name, node)
    if trait.type:
        type_expr = ctx.print_expr(resolve(trait.type, node), path.attr("type"))
        return ast.AnnAssign(var_id, type_expr)
    return var_id
```

**Load / Store** — subscript notation:

```python
def print_load(node, trait, ctx, path):
    source = ctx.print_expr(resolve(trait.source, node), path.attr("source"))
    indices = ctx.print_exprs(resolve(trait.indices, node), path.attr("indices"))
    return ast.Index(source, indices)
```

**Func** — function definition with optional decorator:

```python
def print_func(node, trait, ctx, path):
    # 1. Decorator
    decorator = resolve_kind(trait.text_printer_kind, ctx) if trait.text_printer_kind else None
    # 2. Symbol and params
    symbol = resolve(trait.symbol, node)
    params = resolve(trait.region.def_values, node)
    # 3. Enter scope
    with ctx.new_frame() as frame:
        param_docs = [print_value_def(p, get_trait(p), ctx, ...) for p in params]
        # 4. Prologue
        if trait.text_printer_pre:
            resolve_method(trait.text_printer_pre, node)(ctx, frame)
        # 5. Body
        body = resolve(trait.region.body, node)
        for stmt in (body if isinstance(body, list) else [body]):
            frame.add(print_node(stmt, ctx, ...))
        # 6. Return
        if trait.region.ret:
            ret_val = resolve(trait.region.ret, node)
            frame.add(ast.Return(ctx.print_expr(ret_val, ...)))
    return ast.Function(symbol, param_docs, frame.stmts, decorator=decorator)
```

**For** — for-loop with range bounds:

```python
def print_for(node, trait, ctx, path):
    with ctx.new_frame() as frame:
        # def_values: define loop variable(s)
        loop_var = resolve(trait.region.def_values, node)
        lhs = print_value_def(loop_var, get_trait(loop_var), ctx, ...)
        # Construct iterator expression from range bounds
        if trait.end is not None:
            rhs = build_range_call(
                start=resolve_or_default(trait.start, node, default=0),
                end=resolve(trait.end, node),
                step=resolve_or_default(trait.step, node, default=1),
                kind=resolve_or_default(trait.text_printer_kind, node, default=None),
                attrs=resolve_or_default(trait.attrs, node, default=None),
                ctx=ctx,
            )
        else:
            # Fallback: generic iteration via region.def_expr
            rhs = ctx.print_expr(resolve(trait.region.def_expr, node), ...)
        # body
        body = resolve(trait.region.body, node)
        for stmt in (body if isinstance(body, list) else [body]):
            frame.add(print_node(stmt, ctx, ...))
    return ast.For(lhs, rhs, frame.stmts)

def build_range_call(start, end, step, kind, attrs, ctx):
    """Construct range() or kind(start, end, ...) with default elision."""
    if kind is not None:
        callee = ctx.print_expr(kind)
        args = [ctx.print_expr(start), ctx.print_expr(end)]
        kwargs = {}
        if step != 1:
            kwargs["step"] = ctx.print_expr(step)
        if attrs:
            kwargs["annotations"] = ctx.print_expr(attrs)
        return ast.Call(callee, args, kwargs)
    # Default: Python range() with elision
    args = []
    if start != 0 or step != 1:
        args.append(ctx.print_expr(start))
    args.append(ctx.print_expr(end))
    if step != 1:
        args.append(ctx.print_expr(step))
    return ast.Call(ast.Id("range"), args)
```

**With**, **While**, **If**, **Class** follow the same pattern: resolve
fields from the trait, enter scope, print prologue/body/epilogue.

### Binding Dispatch Rule

When printing an **Assign** whose RHS has a region-based trait (`If`,
`Func`), the printer wraps the assignment around the region:

```python
def print_assign(node, trait, ctx, path):
    rhs_node = resolve(trait.rhs, node)
    rhs_trait = get_trait(rhs_node)

    # Rule: If RHS is an If expression, hoist assignment into branches
    if isinstance(rhs_trait, If):
        return print_assign_if(node, rhs_node, rhs_trait, ctx, path)

    # Rule: If RHS is a Func, print as inline function definition
    if isinstance(rhs_trait, Func):
        return print_func(rhs_node, rhs_trait, ctx, path)

    # Rule: If RHS is void, print as expression statement
    if is_void(rhs_node):
        return ast.ExprStmt(ctx.print_expr(rhs_node, ...))

    # Default: lhs = rhs with optional type annotation
    lhs = print_value_def(resolve(trait.def_values, node), ...)
    rhs = ctx.print_expr(rhs_node, ...)
    return ast.Assign(lhs, rhs, annotation=...)
```

This handles Relax's VarBinding context-dependency (binding.cc:59-82)
without special traits.

### Annotation Intelligence

The printer may omit type annotations when they are inferable from
context.  This is a **printer-level** feature, not a trait parameter:

- Annotations on `call_tir`/`call_dps_packed` bindings: from `sinfo_args`
- Annotations on `Tuple` construction: from field types
- Annotations on `TupleGetItem`: from parent tuple type

The printer exposes a `should_annotate(var, rhs) -> bool` hook that
language modules can customize.

## Parser

### Value-Driven Dispatch

The parser evaluates expressions first, then dispatches on the result:

```
Parser sees:    with T.block("name") as [vi, vj]: ...
Step 1: Eval    T.block("name")  →  BlockFrame
Step 2: Call    BlockFrame.__ffi_text_parse__(parser_ctx, with_node)
```

Each language-module entry returns a **surface object** whose
`__ffi_text_parse__` handles the surrounding AST context.

### Surface Objects

Surface objects expose **position-specific** parse methods:

```python
class SurfaceObject:
    """Returned by language module entries (T.serial, R.dataflow, etc.)."""

    def __ffi_text_parse__(self, parser, node):
        """Generic dispatch.  Called by the parser core."""
        if isinstance(node, ast.For):
            return self.parse_for(parser, node)
        elif isinstance(node, ast.With):
            return self.parse_with(parser, node)
        elif isinstance(node, ast.Assign):
            return self.parse_assign(parser, node)
        elif isinstance(node, ast.FunctionDef):
            return self.parse_function(parser, node)
        elif isinstance(node, ast.ClassDef):
            return self.parse_class(parser, node)
        elif isinstance(node, ast.Expr):
            return self.parse_expr_stmt(parser, node)
        raise ParseError(f"Unexpected position for {type(self)}")
```

This replaces v0's single opaque `__ffi_text_parse__` method.  Benefits:
- `T.launch_thread` can implement `parse_assign`, `parse_expr_stmt`,
  and `parse_with` to support its multi-position syntax
- Unsupported positions fail with a clear error
- Auto-generation is straightforward per position

### Parser Core

```python
class IRParser:
    def __init__(self):
        self.var_table = VarTable()
        self.syntax_ctx = SyntaxContext()

    # --- Value-driven visitors ---

    def visit_For(self, node):
        return self._dispatch(self.eval_expr(node.iter), node)

    def visit_With(self, node):
        if len(node.items) == 1:
            return self._dispatch(
                self.eval_expr(node.items[0].context_expr), node)
        return self._multi_with(node)

    def visit_Assign(self, node):
        if _is_subscript_lhs(node):
            return self._subscript_store(node)
        return self._dispatch_or_bind(self.eval_expr(node.value), node)

    def visit_AnnAssign(self, node):
        return self._dispatch_or_bind(self.eval_expr(node.value), node)

    def visit_FunctionDef(self, node):
        for dec in node.decorator_list:
            if r := self._try_dispatch(self.eval_expr(dec), node):
                return r
        raise ParseError("Unrecognized decorator")

    def visit_ClassDef(self, node):
        for dec in node.decorator_list:
            if r := self._try_dispatch(self.eval_expr(dec), node):
                return r
        raise ParseError("Unrecognized decorator")

    def visit_Expr(self, node):
        self._try_dispatch(self.eval_expr(node.value), node)

    # --- Structural visitors (SyntaxContext) ---

    def visit_If(self, node):     return self.syntax_ctx.handle_if(self, node)
    def visit_While(self, node):  return self.syntax_ctx.handle_while(self, node)
    def visit_Return(self, node): return self.syntax_ctx.handle_return(self, node)
    def visit_Assert(self, node): return self.syntax_ctx.handle_assert(self, node)
    def visit_AugAssign(self, node): ...  # desugar to load + op + store
    def visit_Continue(self, node):  ...
    def visit_Break(self, node):     ...
    def visit_Pass(self, node):      pass

    # --- Dispatch helpers ---

    def _dispatch(self, val, node):
        if hasattr(val, "__ffi_text_parse__"):
            return val.__ffi_text_parse__(self, node)
        raise ParseError(f"Expected IR construct, got {type(val)}")

    def _dispatch_or_bind(self, val, node):
        if hasattr(val, "__ffi_text_parse__"):
            return val.__ffi_text_parse__(self, node)
        self._default_bind(val, node)

    def _try_dispatch(self, val, node):
        if hasattr(val, "__ffi_text_parse__"):
            return val.__ffi_text_parse__(self, node)
        return None
```

### SyntaxContext

Structural constructs (`if`, `while`, `return`, `assert`) have
dialect-specific semantics.  `SyntaxContext` is set by the **language
module**, not derived from traits:

```python
@dataclass
class SyntaxContext:
    handle_if: Callable
    handle_while: Callable
    handle_return: Callable
    handle_assert: Callable
```

The function decorator's surface object pushes a SyntaxContext when
entering the function body:

```python
class PrimFuncDecorator(SurfaceObject):
    def parse_function(self, parser, node):
        with parser.set_syntax_ctx(TIR_SYNTAX):
            params = self._parse_params(parser, node)
            body = parser.visit_body(node.body)
            return PrimFunc(params=params, body=body, ...)
```

Each language module declares its SyntaxContext:

```python
TIR_SYNTAX = SyntaxContext(
    handle_if=tir_handle_if,         # static branching for bool, IR IfThenElse for PrimExpr
    handle_while=tir_handle_while,   # standard while loop
    handle_return=tir_handle_return,  # T.ret()
    handle_assert=tir_handle_assert,  # structured assert
)

RELAX_SYNTAX = SyntaxContext(
    handle_if=relax_handle_if,       # requires else, constructs relax.If, binds result
    handle_while=lambda p, n: raise_(ParseError("while not supported in Relax")),
    handle_return=relax_handle_return,  # R.func_ret_value()
    handle_assert=relax_handle_assert,
)
```

This replaces TVM's dialect-token dispatch stack.

### Parser Generation

The framework auto-generates surface objects from traits.  For each
language module entry, the trait metadata determines the parse logic:

```python
# ffi.lang.for_iter(TIRFor, kind="serial") generates:
class _SerialForIter(SurfaceObject):
    def __init__(self, *args, **kwargs):
        # args/kwargs carry the range call arguments: (start, end, step=...)
        self._start, self._end = parse_range_args(args, kwargs)
        self._step = kwargs.get("step", None)
        self._annotations = kwargs.get("annotations", None)

    def parse_for(self, parser, node):
        with parser.var_table.frame():
            loop_var = parser.create_var(node.target)
            parser.var_table.define(node.target.id, loop_var)
            body = parser.visit_body(node.body)
            return TIRFor(loop_var=loop_var,
                          min=self._start, extent=self._end - self._start,
                          step=self._step, kind=ForKind.SERIAL,
                          annotations=self._annotations, body=body)

# ffi.lang.with_ctx(SBlock) generates:
class _SBlockFrame(SurfaceObject):
    def parse_with(self, parser, node):
        with parser.var_table.frame():
            as_vars = self._create_iter_vars()
            if node.items[0].optional_vars:
                parser.bind_vars(node.items[0].optional_vars, as_vars)
            body = parser.visit_body(node.body)
            return SBlock(iter_vars=as_vars, body=body, ...)

# ffi.lang.decorator(PrimFunc) generates:
class _PrimFuncDecorator(SurfaceObject):
    def parse_function(self, parser, node):
        with parser.var_table.frame():
            with parser.set_syntax_ctx(TIR_SYNTAX):
                params = self._parse_params(parser, node)
                body = parser.visit_body(node.body)
                return PrimFunc(params=params, body=body, ...)
```

## Language Modules

Each dialect provides a language module — the namespace in printed text:

```python
@ffi.lang("T")
class T:
    prim_func    = ffi.lang.decorator(PrimFunc)
    serial       = ffi.lang.for_iter(TIRFor, kind="serial")
    parallel     = ffi.lang.for_iter(TIRFor, kind="parallel")
    block        = ffi.lang.with_ctx(SBlock)
    match_buffer = ffi.lang.binding(MatchBufferRegion)
    Buffer       = ffi.lang.proxy(BufferProxy)
    int32        = ffi.lang.dtype("int32")
    syntax       = TIR_SYNTAX
```

Each entry returns a surface object with position-specific parse methods,
auto-generated from the trait on the target IR type.  `.pyi` stubs via
`tvm-ffi-stubgen`.

## Canonical Syntax

Each trait maps to a well-known Python syntax pattern.  This is the
**stable, roundtrip-safe form** that the auto-generated printer produces:

```python
# Value reference
${name}

# BinOp / UnaryOp
${lhs} + ${rhs}
-${operand}

# Load / Store
${source}[${indices}]
${target}[${indices}] = ${value}

# Assign
${def_values} = ${rhs}
${def_values}: ${type} = ${rhs}

# Return
return ${value}

# Func
@${text_printer_kind}
def ${symbol}(${def_values}):
    ${prologue}
    ${body}

# For (range-based)
for ${def_values} in range(${start}, ${end}, ${step}):
    ${body}
# For (with kind)
for ${def_values} in ${text_printer_kind}(${start}, ${end}, step=${step}):
    ${body}
# For (generic, via region.def_expr)
for ${def_values} in ${def_expr}:
    ${body}

# With
with ${def_expr} as ${def_values}:
    ${prologue}
    ${body}

# If
if ${cond}:
    ${then_body}
else:
    ${else_body}

# While
while ${cond}:
    ${body}

# Class
@${text_printer_kind}
class ${symbol}:
    ${prologue}
    ${members}
```

### Sugar vs Canonical

Traits generate **canonical** syntax.  Alternative spellings are surface
sugar, handled by Level 2 overrides or surface codecs:

| Sugar | Why not canonical |
|-------|-------------------|
| `T.grid(4, 4)` | Desugared to nested serial loops in final IR |
| `T.axis.remap("SSR", ...)` | Desugared to individual axis bindings |
| `T.macro` / `R.macro` | Macro body inlined; definition lost |
| `T.meta_var` | Parse-time-only; not in final IR |
| `R.emit_te` | Desugared to `call_tir` |
| `T.comm_reducer(lambda ...)` | Lambda lowered away |

Sugar is supported via Level 2 overrides and is never required for the
canonical printer to work.

## Module Parsing

Module-level classes (`@I.ir_module`) require multi-pass parsing.  This
is handled by the module decorator's surface object, not by the Class
trait alone:

```python
class IRModuleDecorator(SurfaceObject):
    def parse_class(self, parser, node):
        # Pass 1: module attrs, global infos, extern members
        for stmt in node.body:
            if is_non_function(stmt):
                self._parse_preamble(parser, stmt)
        # Pass 2: forward-declare all functions
        for stmt in node.body:
            if isinstance(stmt, ast.FunctionDef):
                gvar = self._declare_function(parser, stmt)
                parser.var_table.define(stmt.name, gvar)
        # Pass 3: parse function bodies
        for stmt in node.body:
            if isinstance(stmt, ast.FunctionDef):
                self._parse_function(parser, stmt)
        return IRModule(...)
```

This is Level 2 code on the surface object, not on the IR class.

## Comparison with Existing Systems

### vs TVM TVMScript (~5000 LOC per dialect)

TVMScript requires:
- Hand-written C++ printer per node type (IRDocsifier dispatch)
- Hand-written Python parser per node type (dispatch.register handlers)
- Global dispatch table + isinstance chains (bind_assign_value)
- Dialect token stack management

This design replaces:
- **Dispatch table** → value-driven dispatch via `__ffi_text_parse__`
- **isinstance chains** → each type's surface object handles its own parsing
- **Token stack** → SyntaxContext set by function decorator
- **C++ printer boilerplate** → auto-generated from traits

**Estimated reduction**: Level 0+1 nodes need zero printer/parser code.
Level 2 nodes need ~50% less code (framework handles scope, variable
management, frame stacking).  For a TIR-scale dialect (~32 nodes):
~15 Level 1 + 8 Level 0 + 9 Level 2 = ~70% code reduction.

### vs MLIR ODS

| Aspect | MLIR ODS | This Design |
|--------|----------|-------------|
| Definition language | TableGen | Python decorators |
| Target format | MLIR assembly | Python-style text |
| Auto-generation | `assemblyFormat` DSL | Trait + `text_*` params |
| Fallback | `hasCustomAssemblyFormat` | Level 2 override |
| Fallback rate | ~30% of operations | ~28% of TIR nodes |
| Trait system | C++ traits (SingleBlock, etc.) | Python traits (For, If, etc.) |
| Region model | Explicit block arguments + terminators | Python scoping + def_values |

MLIR ODS is more expressive for MLIR assembly (optional groups, oilist,
custom directives).  This design is more natural for Python-style text
(Python syntax IS the format).  They serve different format families
and complement each other.
